## Work Completed

**Dashboard consistency fixes**
- Root-caused a "Pending page shows 16, Pendentes Abertas KPI shows 41" mismatch: the KPI read `getRiskMetrics().openCount` (includes live/already-kicked-off bets), the Pending page read `getPendingRows()` (kickoff-future only). Fixed the KPI to call `getPendingRows()` directly.
- Same root cause reproduced in the "Muitas apostas abertas" alert (`computeAlerts()`); fixed identically.
- Verified exposure/risk widgets (`stakeOpen`, `potencialLucro`, "Risco Atual") deliberately still use `getRiskMetrics().openCount` — different metric (total capital at risk, includes live bets), left untouched.

**Manual bet / bankroll cloud synchronization investigation (LIVE-1 and follow-ons)**
- Full code audit of the bankroll movement persistence pipeline, requested and delivered before any change (in-memory shape, save serialization, cloud schema, every consumer, recovery paths).
- Root cause found: `_doLoadCloudState()` and `_reloadManualBetsFromCloud()` never copied `content.movements` into `state.movements`, so any fresh session (e.g. Incognito) silently ignored all bankroll deposits/withdrawals while every other figure recovered correctly.
- Fixed both functions to assign `state.movements = Array.isArray(content.movements) ? content.movements : []`, matching how every other recovered field is handled. No new save call added — the existing post-recovery `saveLocalState()` calls now persist movements too.
- Verified end-to-end in a real headless browser: a "Normal" window (localStorage pre-seeded) and a fresh "Incognito" window (forced cloud recovery) both fed the identical mocked cloud payload now render byte-identical bankroll, KPIs, exposure figures, and Movement History tables.

**Railway `GITHUB_REPO` misconfiguration (discovered during verification, not part of the original report)**
- While verifying against live production, `GET /load` was found returning an empty `{}`. Traced to `sync_server.py`'s `GITHUB_OWNER`/`GITHUB_REPO` environment variables — Railway's `GITHUB_REPO` held the fully-qualified `jorgepita/apostas-over-futebol` instead of just the repo name, producing a doubled, 404-ing GitHub Contents API path.
- Confirmed `update_results.py` was unaffected (hardcodes the same two values as constants, doesn't read them from the environment) — this is why GitHub Actions settlement and `/run-settlement` kept working the whole time.
- Recommended fixing the Railway env var rather than changing code (per ADR-010). The variable was corrected outside this session (observed already fixed when re-checked); no code change was made for this issue.

**Cleanup phase (this request)**
- Removed all temporary diagnostic instrumentation added during the investigation: `debugger` statements, `[F1]`–`[F6]`, `[BOOT-DIAG]`, `[RECOVERY]`, `[RECOVERY-SCHEMA]` console logging in `index.html`; `[STEP 1]`–`[STEP 5]` print statements in `update_results.py`. Normal production logging (`[settlement] ...`, `console.error` catch handlers, the feature-flagged `diag_log()` helper) left untouched.
- Verified `python3 -m py_compile update_results.py` and `node --check` on the extracted `index.html` script block both pass after cleanup.
- Reran the full Normal-vs-Incognito browser verification after cleanup (see Validation Performed) to confirm the fixes still hold with instrumentation removed.
