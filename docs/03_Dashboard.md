# Dashboard

This document describes the Football Bot dashboard in technical detail. For system architecture, data flows, and component interactions, see `01_Architecture.md` and `02_Data_Flow.md`. This document focuses exclusively on the frontend.

---

## 1. Dashboard Overview

`index.html` is a single self-contained HTML file. It is the operational interface for the entire betting system: it displays picks, manages manual bets through their complete lifecycle, presents analytics, and controls bankroll settings.

**The dashboard owns:**
- The in-memory `state` object and its rendering pipeline.
- The `localStorage` copy of `cloud_state.json`.
- The user's interaction with manual bets (create, approve, reject, settle manually).

**The dashboard does not own:**
- Any file on GitHub. It cannot commit to the repository.
- The canonical state of manual bets — that belongs to `cloud_state.json` on GitHub. `localStorage` is a working copy.
- Settlement logic. Settlement runs on the backend (`update_results.py`).
- Pick generation. Picks are generated by the Python pipeline and consumed as CSVs.

**Language:** All user-facing strings are in Portuguese (PT-PT).

## 2. Application Structure

The application is a vanilla JavaScript SPA inside a single HTML file. There is no framework, no build step, and no module system. All code runs in a single `<script>` block at the bottom of the `<body>`.


Module-level flags:

| Flag | Purpose |
|---|---|
| `autoSaveInFlight` | True while a `POST /save` is in progress; prevents concurrent saves |
| `hasPendingCloudChanges` | True if local state has not yet been saved to the cloud |
| `initialLoadDone` | True after the first `loadData()` completes |
| `_autoSaveTimer` | Debounce handle for the 4-second auto-save |
| `_dirtyGeneration` | Counter that increments on every `markDirty()`; detects in-flight changes |
| `_settlementRunning` | True while a settlement request is pending |
| `_settlementMessage` | Status text shown on the Run Settlement button |

### Rendering pipeline

```
User action / interval tick
        │
        ▼
  state mutation
        │
        ▼
  markDirty()          ← schedules cloud save (4s debounce)
  saveLocalState()     ← writes localStorage synchronously
        │
        ▼
  rerenderAll()        ← re-renders every panel
        │
        ▼
  setActiveTab()       ← ensures the active tab is visible
```

### Event handling

Event listeners are attached once in `bindStaticButtons()`, `bindStaticInputs()`, `bindTabButtons()`, and `bindManualControls()`. All use event delegation on stable parent elements to avoid rebinding after re-renders. `rerenderAll()` calls `bindTabButtons()` and `bindManualControls()` at the end to attach handlers to newly rendered table rows.

## 3. Global State

The `state` object is the single source of truth for all in-memory data. It is initialised with empty/null values and populated by `loadLocalState()` and `loadData()` during `boot()`.

```javascript
const state = {
  footballDaily:     [],    // today's bot picks (merged from daily + BTTS CSVs)
  footballHistory:   [],    // all bot picks history (picks_history.csv)
  leagueStats:       [],    // per-league stats (league_stats.csv)
  bankrollInicial:   null,  // initial bankroll amount (number or null)
  bankrollInicialSet: false,// true once bankroll is locked
  localEdits:        {},    // pickKey → {apostada, oddReal, stakeReal, resultadoManual}
  manualBets:        [],    // manual bets (cloud_state.json source)
  movements:         [],    // bankroll deposits and withdrawals
  sessionStartDate:  '',    // ISO date when current betting session started
  lastUpdated:       null,  // Date object of last loadData() call
  activeTab:         'tab-summary',
  mobileActionsOpen: false,
  cloudAvailable:    false  // true once a cloud round-trip succeeds
};
```

Runtime-only additions (not persisted to localStorage):

| Property | Set by | Purpose |
|---|---|---|
| `state.manualBetsRemote` | `loadData()` | Rows from `manual_bets.csv` (always empty) |
| `state.fixtures` | `loadData()` | Rows from `fixtures_today.csv` (kickoff lookup for Scout) |

### Important state properties

**`state.manualBets`**  
Array of manual bet objects loaded from localStorage at startup. Each object has: `id`, `data`, `liga`, `jogo`, `mercado`, `odd`, `stake`, `resultado`, `status` (`"pending"` / `"approved"` / `"settled"`), `notas`, and optional analysis fields (`botOpinion`, `edgeAtAnalysis`, `analysisSnapshot`).  
Updated by: user actions (create, approve, reject), `_doLoadCloudState()`, `_reloadManualBetsFromCloud()`.  
Persisted by: `saveLocalState()` to localStorage; `saveCloudState()` to `cloud_state.json` via Railway.

**`state.footballDaily`**  
Array of today's bot pick rows. Populated by `loadData()` by merging `picks_hoje_github.csv` and `picks_btts.csv`. Each row is a normalised bot CSV row. Rows with missing `Data`, `Jogo`, or invalid `Odd` are filtered out.  
Used by: Daily Picks page, Pending page (for kickoff lookup), Live Center (bot rows).

**`state.footballHistory`**  
Array of all rows from `picks_history.csv`. Includes both settled and unsettled picks.  
Used by: History page, Analytics, bot vs manual comparison, bankroll calculations.

**`state.localEdits`**  
Map from `pickKey` (string built from `Data||Liga||Jogo||Mercado`) to an edit object `{apostada: bool, oddReal: string, stakeReal: string, resultadoManual: string}`. Records user-entered execution data for bot picks.  
Updated by: user editing "Apostada", "Odd Real", "Stake Real" fields in the History page.  
Merged into bot pick rows by: `getRowWithLocalEdits(row)` before display and analytics.

**`state.bankrollInicial`** and **`state.bankrollInicialSet`**  
Initial bankroll amount. Once `bankrollInicialSet` is true, the bankroll input is locked and cannot be changed. This prevents accidental resets.  
Used by: all profit/ROI calculations, bankroll page, Kelly stake display.

**`state.movements`**  
Array of bankroll deposit and withdrawal records `{id, date, type, amount, note}`.  
Used by: Bankroll page (movements table, evolution chart).

**`state.sessionStartDate`**  
ISO date string (YYYY-MM-DD) when the current betting session started. Used to scope analytics to the current session.

**`state.cloudAvailable`**  
Boolean reflecting whether the last cloud operation succeeded. Drives the cloud status indicator in the sidebar.

```
          │   │   ├─ fetch manual_bets.csv          → state.manualBetsRemote (empty)
          │   │   └─ fetch fixtures_today.csv       → state.fixtures (kickoff data)
          │   └─ loadLeagueStats():
          │       └─ fetch league_stats.csv         → state.leagueStats
          │
          ├─ checkPendingAlerts()
          │   └─ sends Telegram alert if any active bot pick is ≤ 30 min to kickoff
          │
          ├─ if (!hasMeaningfulLocalState()):
          │   └─ _doLoadCloudState({ fromUser: false })
          │       ├─ GET Railway /load → cloud_state.json
          │       ├─ state.manualBets ← content.manualBets
          │       ├─ state.bankrollInicial, localEdits, sessionStartDate ← cloud

          │

                loadData(true);
                checkPendingAlerts();

```

**Auto-recovery guard:** `hasMeaningfulLocalState()` returns `true` if `bankrollInicialSet` is true, `localEdits` has entries, `manualBets` is non-empty, or `movements` is non-empty. When `true`, the cloud auto-recovery step is skipped. The user's existing localStorage session takes precedence.

## 5. Rendering Architecture

### `rerenderAll()`

The primary render function. Called after any data change. Re-renders every panel regardless of which tab is active, then calls `setActiveTab()` to show the correct panel.

```javascript
function rerenderAll() {
  fillFilters();
  rerenderSummaryOnly();     // Home / Summary tab
  rerenderDayOnly();         // Daily Picks tab
  rerenderHistoryOnly();     // History tab
  rerenderManualOnly();      // Manual Bets + Pending + Live Center
  rerenderLiveOnly();        // Live Center (again, for safety)
  renderPicksKpiRow();       // KPI strip for Daily Picks
  renderHistoryKpiRow();     // KPI strip for History
  renderAnalyticsPerformers(); // Per-league analytics table
  renderBankrollPage();      // Bankroll / Settings tab
  renderVersus();            // Bot vs Manual tab
  renderLeagueAnalytics();   // League statistics table
  // bind UI controls, update buttons, set active tab
}
```

Every render call is wrapped in a try-catch. A failure in one panel does not prevent others from rendering.

### Partial renderers

Partial re-renders are used when only one area needs to update (e.g. after a Live Center settle action):

| Function | Renders |
|---|---|
| `rerenderSummaryOnly()` | Home tab summary cards |
| `rerenderDayOnly()` | Daily Picks table, KPI row |
| `rerenderHistoryOnly()` | History table, KPI row, intelligence panels, timeline, equity curve |
| `rerenderManualOnly()` | Manual Bets table, Scout, Pending queue, Live Center |
| `rerenderLiveOnly()` | Live Center only (wrapped in try-catch; swallows errors silently) |

### Tab activation

`setActiveTab(tabId)` shows the tab panel matching `tabId` and hides all others. It also calls a targeted renderer for interactive tabs that need fresh state immediately:

- `tab-summary` → `renderMobileHomeDash()`
- `tab-day` → `renderFootball()`
- `tab-pending` → `renderPendingQueue()`
- `tab-live` → `renderLiveCenter()`

Tab switching does not trigger `loadData()` or any network call.

### Auto-save and dirty state

After any state mutation:
1. `markDirty()` is called.
2. `hasPendingCloudChanges` is set to `true`.
3. The cloud status indicator updates to "alterações por guardar".
4. A 4-second debounce timer is set. If no further changes arrive, `saveCloudState()` runs.
5. `saveCloudState()` sends `POST /save` with the full `state` object.
6. On success, `hasPendingCloudChanges` is cleared (only if `_dirtyGeneration` has not changed during the fetch).
7. On failure, `hasPendingCloudChanges` remains `true` and a retry is scheduled.

## 6. Page-by-Page Documentation

### Home / Summary (`tab-summary`)

**Purpose:** At-a-glance overview on mobile. On desktop, this tab is also visible but the primary navigation is via the sidebar.

**Data sources:** `state.footballDaily` (today's picks count), `state.footballHistory` + `state.localEdits` (settled history for P&L summary), `state.manualBets` (manual bet counts).

**User interactions:** None. Read-only summary cards.

**Update triggers:** Any `rerenderAll()` call; tab activation on mobile.

---

### Daily Picks (`tab-day`)

**Purpose:** Shows today's bot picks that have not yet been marked as executed (`apostada = false`). Once the user marks a pick as "Apostada" (bet placed), it disappears from this view.

**Data sources:** `state.footballDaily` — merged from `picks_hoje_github.csv` (O2.5 picks) and `picks_btts.csv` (BTTS picks). Rows are filtered to remove already-approved picks (`_apostadaBool === true`).

**User interactions:**
- Filter by market (O2.5, BTTS), league, date.
- Mark a pick as "Apostada" — sets `localEdits[pickKey].apostada = true`, calls `markDirty()`, `saveLocalState()`, `rerenderAll()`.

**Update triggers:** `loadData()` (page load, 60-second interval); any `rerenderAll()`.

**Dependencies:** `getRowWithLocalEdits()` merges `state.localEdits` into each bot row before display.

---

### Live Center (`tab-live`)

**Purpose:** Shows all active bets — both bot picks and manual bets — that are in play right now (kickoff has passed, result not yet settled).

**Data sources:**
- Manual rows: `getManualRowsMerged()` filtered for `isLocal === true`, `status === 'approved'`, resultado not in `{W, L, P}`, and kickoff ≤ now (or date ≤ today if no kickoff data).
- Bot rows: `state.footballDaily` rows where `apostada === true` and result not settled, or history rows for today.

**User interactions:**
- "Run Settlement" button — `POST /run-settlement` to Railway; calls `_reloadManualBetsFromCloud()` on response.
- Manual settle buttons (W/L/P) — marks the bet as settled locally; calls `markDirty()`.

**Update triggers:** Any `rerenderAll()` or `rerenderLiveOnly()` call.

**`getLiveRows()` filter (manual bets):**
```javascript
b.isLocal === true
&& b.status === 'approved'
&& !['W','L','P'].includes(b._resultKey)
&& b.data exists
&& (kickoffMs !== null ? kickoffMs <= Date.now() : b.data <= today)
```

---

### Pending (`tab-pending`)

**Purpose:** Shows approved manual bets whose kickoff has not yet arrived. These bets are ready to watch but not yet "live".

**Data sources:** `getManualRowsMerged()` filtered for `isLocal === true`, `status === 'approved'`, and kickoff > now (or date > today if no kickoff data).

**`getPendingRows()` filter:**
```javascript
b.isLocal === true
&& b.status === 'approved'
&& (kickoffMs !== null ? kickoffMs > Date.now() : b.data > today)
```

**User interactions:**
- "Cancelar" button — rejects a pending bet; sets status back to a rejected state, calls `markDirty()`.

**Update triggers:** Any `rerenderAll()` or `rerenderManualOnly()`.

---

### Manual Bets (`tab-manual`)

**Purpose:** Two functions in one tab:
1. **Scout Workspace** — search for fixtures, run the Poisson model, create manual bets.
2. **Bet list** — list of all manual bets not yet approved, with approve/reject actions.

**Data sources:**
- Scout: API-Football `/fixtures` endpoint called on-demand when the user searches.
- Bet list: `state.manualBets` filtered for `status === 'pending'`.

**User interactions:**
- Type a team name or date → `renderManualScout()` queries API-Football.
- Select a fixture → Poisson model runs in JavaScript → edge and Kelly stake are computed.
- "Criar Aposta" (Create Bet) → `addManualBet()` — pushes to `state.manualBets`, calls `markDirty()`, navigates to `tab-manual`.
- "Aprovar" (Approve) → sets `status: 'approved'`, calls `markDirty()`. Bet moves to Pending/Live.
- "Rejeitar" (Reject) → removes from `state.manualBets`, calls `markDirty()`.
- Manual entry form — allows creating a bet without Scout by filling fields directly.

**Update triggers:** Any `rerenderAll()` or `rerenderManualOnly()`.

---

### History (`tab-history`)

**Purpose:** Full settled bet history — both bot picks (with user execution data from `localEdits`) and manual bets with results.

**Data sources:**
- Bot picks: `state.footballHistory` merged with `state.localEdits` via `getRowWithLocalEdits()`.
- Manual bets: `state.manualBets` filtered for `resultado` in `{W, L, P}`.
- `getHistoryRowsMerged()` produces a unified list of settled rows from both sources.

**User interactions:**
- Filter by date range, league, market, result.
- Edit "Odd Real", "Stake Real" for bot picks — stored in `state.localEdits`, calls `markDirty()`.
- Export filtered history as CSV via `exportRealCsv()`.

**Update triggers:** `loadData()` (history CSV refresh); any `rerenderAll()` or `rerenderHistoryOnly()`.

**Sub-panels in this tab:**
- KPI strip — total stake, total profit, ROI, win rate.
- History Intelligence — streak analysis, best/worst market.
- Timeline — chronological pick list with result colours.
- Equity Curve — cumulative profit over time (inline ASCII-style or SVG chart).

---

### Analytics (`tab-analytics`)

**Purpose:** Deeper performance intelligence. Multiple sub-sections using the settled history.

**Data sources:** `getFilteredRealClosedRows(getSummaryFilters())` — the settled bot picks with execution data. `getResolvedManualBets()` — settled manual bets.

**Sub-sections:**
- **League Analytics** — per-league ROI, win rate, avg Kelly, total picks. Enriched with `state.leagueStats`.
- **Analytics Performers** — top and bottom leagues by ROI.
- **Bot vs Manual (`tab-versus`)** — side-by-side comparison of real bot pick performance vs manual bet performance. Stats: total stake, profit, ROI, win rate, best/worst league.

**Update triggers:** Any `rerenderAll()`.

---

### Bankroll / Settings (`tab-bankroll`)

**Purpose:** Bankroll management and configuration.

**Data sources:** `state.bankrollInicial`, `state.movements`, `state.footballHistory`, `state.localEdits`, `state.manualBets`.

**Sub-sections rendered by `renderBankrollPage()`:**
- `renderBankrollOverview()` — current balance = initial bankroll + net P&L.
- `renderBankrollComposition()` — split between bot picks and manual bets.
- `renderBankrollPerformanceBreakdown()` — P&L breakdown by market.
- `renderBankrollEvolution()` — balance over time.
- `renderMovementsTable()` — list of deposits and withdrawals; add/remove actions.
- `renderBankrollConfig()` — Kelly fraction, edge thresholds (read from `state.localEdits` or defaults).
- `renderBankrollAudit()` — reconciliation table.

**User interactions:**
- Set initial bankroll — locked once set (`bankrollInicialSet`).
- Add deposit/withdrawal — `state.movements.push({...})`, calls `markDirty()`.
- Load Cloud / export buttons.

**Update triggers:** Any `rerenderAll()`.

## 7. Manual Bets Workspace

The complete lifecycle of a manual bet in the dashboard:

### Step 1 — Fixture Search (Scout)

The user types a team name or date into the Scout search field. `renderManualScout()` calls the API-Football fixtures endpoint. Results are shown as fixture cards with team names, league, date, and kickoff.

### Step 2 — Poisson Analysis

The user clicks a fixture card. The Poisson model runs in JavaScript:
- Fetches `state.fixtures` (from `fixtures_today.csv`) to get `λ_home` and `λ_away`.
- Computes goal probabilities for O1.5, O2.5, O3.5, BTTS.
- Compares model probability to user-entered odd.
- Displays edge, Kelly fraction, recommended stake, and a verdict (value / no value).

Analysis result is stored in the bet object as `botOpinion`, `edgeAtAnalysis`, `scoreAtAnalysis`, `analysisSnapshot`.

### Step 3 — Bet Creation

The user fills in market, odd, stake (pre-filled from Scout analysis if applicable) and clicks "Criar Aposta".

`addManualBet()` is called:
```javascript
state.manualBets.push({
  id: makeManualId(),       // unique ID: timestamp + random suffix
  data, liga, jogo, mercado, odd, stake, resultado: '',
  status: 'pending',        // initial state
  analysisStale: false,
  // analysis fields if Scout was used
});
markDirty();               // schedules cloud save
```

The bet is immediately visible in the Manual Bets list with status "pending".

### Step 4 — Approval

The user clicks "Aprovar" (Approve) on the bet in the Manual Bets list.

```javascript
state.manualBets[idx].status = 'approved';
markDirty();
rerenderAll();
```

The cloud auto-save fires within 4 seconds. The bet transitions:
- Removed from the Manual Bets "pending" list.
- Appears in **Pending** (if kickoff has not yet passed).
- Appears in **Live Center** (if kickoff has passed).

### Step 5 — Rejection

The user clicks "Rejeitar" (Reject). The bet is spliced from `state.manualBets`. `markDirty()` is called. The bet is gone from all views and from `cloud_state.json` on next save.

### Step 6 — Persistence

After every create/approve/reject:
1. `saveLocalState()` writes to localStorage synchronously.
2. `markDirty()` schedules `saveCloudState()` after 4 seconds.
3. `saveCloudState()` sends `POST /save` to Railway with the full `state` object.
4. Railway writes `cloud_state.json` to GitHub.

The cloud status indicator reflects the save state in real-time.

### Step 7 — Live Center

After kickoff passes, `getLiveRows()` includes the bet. It appears in the Live Center alongside any active bot picks.

### Step 8 — Settlement

Settlement can be triggered:
- **From the dashboard:** User clicks "Run Settlement" → `POST /run-settlement` → Railway runs `run_settlement_remote()` → result written to `cloud_state.json` → `_reloadManualBetsFromCloud()` pulls the result back into `state.manualBets` → `rerenderAll()`.
- **Externally (GitHub Actions):** `cloud_state.json` is updated on GitHub. The browser is not notified. The user must click "Load Cloud" to see the result.

### Step 9 — History

After settlement, `resultado` is W/L/P. `getManualRowsMerged()` returns the bet with `_resultKey` in `{W, L, P}`. It is excluded from the Live Center and Pending filters. It appears in the History page and feeds Analytics and Bankroll calculations.

### State Transition Summary

```
[created]
   status: 'pending'
   resultado: ''
      │
      │ user clicks Aprovar
      ▼
[approved, pre-kickoff]
   status: 'approved'
   resultado: ''
   → appears in Pending
      │
      │ kickoff time passes
      ▼
[approved, post-kickoff]
   status: 'approved'
   resultado: ''
   → appears in Live Center
      │
      │ settlement writes resultado
      ▼
[settled]
   status: 'approved' (unchanged by settlement)
   resultado: 'W' / 'L' / 'P'
   → appears in History, Analytics, Bankroll
```

## 8. Live Center

### Data sources

`getLiveRows()` builds the unified live row list:

1. **Manual rows:** from `getManualRowsMerged()` with filter:
   ```
   isLocal === true
   AND status === 'approved'
   AND _resultKey not in {W, L, P}
   AND data field exists
   AND (kickoff timestamp ≤ now OR date ≤ today)
   ```

2. **Bot rows:** from `getDailyRowsMerged()` and `state.footballHistory` for rows where `apostada === true` and result is not settled.

Both sets are combined and sorted by date.

### Merge process

`getManualRowsMerged()` merges two sources:
- `remoteRows` from `state.manualBetsRemote` (always empty in production; `manual_bets.csv` has no data): `isLocal: false`.
- `localRows` from `state.manualBets` (cloud-sourced bets): `isLocal: true`.

The Live Center filter is applied to the merged list. Only `isLocal: true` bets pass (`status === 'approved'` is only meaningful on local bets).

### Rendering

`renderLiveCenter()` builds an HTML table. Each row shows: date, league, game, market, odd, stake, time since kickoff.

For manual rows: a quick-settle control (W/L/P buttons) is shown. For bot rows: the row is read-only (settlement runs on the backend).

The "Run Settlement" button is rendered in this panel. Its state is controlled by `_settlementRunning` and `_settlementMessage`.

### Expected lifecycle of a live bet

```
Kickoff passes
   └─ bet appears in Live Center (getLiveRows includes it)
         │
         │ settlement runs (Railway or GitHub Actions)
         ▼
   resultado written to cloud_state.json
         │
         │ user clicks "Load Cloud" or triggers settlement from this session
         ▼
   state.manualBets refreshed
         │
         ▼
   bet disappears from Live Center (resultado now in {W,L,P})
   bet appears in History
```

## 9. History

### History generation

`getHistoryRowsMerged()` combines bot picks and manual bets into a single sorted list:
- Bot picks: from `state.footballHistory` (all rows), merged with `state.localEdits` via `getRowWithLocalEdits()`.
- Manual bets: from `state.manualBets` (those with `resultado` in `{W, L, P}`).

**Filtering** is applied by `getFilteredRealClosedRows(getHistoryFilters())`:
- Date range
- League
- Market (O2.5, BTTS)
- Result (W, L, P)
- Execution status (apostada = true for bot picks)

### Display

`renderClosedRealTable()` renders the filtered rows. Each bot row shows: date, league, game, market, model odd, stake, edge, result, model profit. Editable fields (OddReal, StakeReal) are inline inputs.

### Analytics sub-panels

`rerenderHistoryOnly()` also calls:
- `renderHistoryKpiRow()` — total stake, profit, ROI, win rate KPIs.
- `renderHistoryIntelligence()` — current streak, best/worst run.
- `renderHistoryTimeline()` — pick cards in chronological order, colour-coded by result.
- `renderHistoryEquity()` — cumulative profit chart.

### Export

`exportRealCsv()` downloads a semicolon-delimited CSV of the filtered rows. Columns: `Data;Origem;Liga;Jogo;Mercado;Resultado;Odd;Stake;Lucro`.

## 10. Analytics

### League Analytics

**Data sources:** `state.leagueStats` (from `league_stats.csv`) enriched with actual bet history via `buildAnalytics(rows, 'league')`. The enrichment joins by league name and adds real win rates and P&L from executed picks.

**Displays:**
- Table of all leagues: ROI%, WinRate%, avg Kelly, Tier, total picks.
- Insight cards: best ROI league, highest win rate, highest avg Kelly, most active league.
- Filter by market (O2.5, BTTS) and minimum sample size.

### Analytics Performers

`renderAnalyticsPerformers()` uses `buildAnalytics()` to rank leagues by actual performance. Leagues with fewer than 2 executed bets are excluded.

### Bot vs Manual (`tab-versus`)

`renderVersus()` computes and compares:

| Metric | Bot | Manual |
|---|---|---|
| Source | `getRealResolvedBotHistory()` — executed bot picks with real outcomes | `getResolvedManualBets()` — settled manual bets |
| Stake | Sum of `StakeReal€` | Sum of `stake` |
| Profit | Sum of `LucroReal€` | Sum of `_lucro` |
| ROI | profit / stake × 100 | profit / stake × 100 |
| Win rate | wins / total | wins / total |

Minimum sample threshold: `MIN_SAMPLE_COMPARE = 5` bets required before comparison is shown.

### Bankroll Analytics

`renderBankrollPage()` calls seven sub-renderers:
- **Overview:** current balance = `bankrollInicial` + net P&L from all settled bets + net deposits/withdrawals from `state.movements`.
- **Composition:** split of P&L between bot picks and manual bets.
- **Performance Breakdown:** P&L by market (O2.5 vs BTTS).
- **Evolution:** balance over time using settled dates.
- **Movements table:** chronological list of deposits and withdrawals; add/remove actions.
- **Config:** displays Kelly fraction and edge thresholds (informational; these are set on the backend in `config.json`).
- **Audit:** reconciliation between bankroll formula and actual settled amounts.

## 12. Dashboard Refresh Model

### Automatic refresh — CSV data (60 seconds)

```
setInterval(60s)
  └─ loadData(true)
      ├─ fetches picks_hoje_github.csv
      ├─ fetches picks_btts.csv
      ├─ fetches picks_history.csv
      ├─ fetches manual_bets.csv (empty)
      └─ fetches fixtures_today.csv
      → state.footballDaily, state.footballHistory updated
      → rerenderAll()

  └─ checkPendingAlerts()
      └─ sends Telegram if any bot pick ≤ 30 min to kickoff
```

Does **not** refresh `state.manualBets`. Manual bet state is not part of the 60-second cycle.

### Automatic cloud save (4-second debounce)

```
Any state mutation
  └─ markDirty()
      └─ setTimeout(4000ms)
          └─ saveCloudState() → POST /save → Railway → GitHub
```

If a new mutation arrives within the 4 seconds, the timer resets. Only one save fires per burst of edits.

### Manual cloud reload

```
User clicks "Load Cloud"
  └─ _doLoadCloudState({ fromUser: true })
      └─ GET /load → Railway → GitHub → cloud_state.json
      └─ state.manualBets, bankrollInicial, localEdits, sessionStartDate ← cloud
      └─ saveLocalState()
      └─ rerenderAll()
```

Season recency guard: if the cloud `sessionStartDate` is older than the local one, the load is blocked with an alert.

### Post-settlement cloud reload

```
User clicks "Run Settlement"
  └─ POST /run-settlement → Railway (synchronous, up to ~90s)
      └─ settlement result returned
  └─ _reloadManualBetsFromCloud()
      └─ GET /load → Railway → cloud_state.json
      └─ state.manualBets ← content.manualBets
      └─ saveLocalState()
      └─ rerenderAll(false)
```

This is the only automatic cloud refresh that is not user-initiated. It happens only when settlement is triggered from the current browser session.

### Rendering refresh

`rerenderAll()` is called:
- After `loadData()` completes (every 60 seconds).
- After any `state` mutation via user action.
- After `_doLoadCloudState()` or `_reloadManualBetsFromCloud()` completes.
- After tab activation for Live Center, Daily Picks, and Pending (targeted renderer only).
- On window resize (debounced to 150ms).

## 13. Error Handling

### Network failures — CSV fetch

`loadData()` wraps all fetches in a try-catch. If `picks_history.csv` fails to load, the entire `loadData()` call fails (it is the required file; all others are optional).

`_loadDataFailCount` tracks consecutive failures. `_setFetchBanner()` shows a red banner when count ≥ 1:
- 1–2 failures: "Última atualização falhou. Nova tentativa em breve."
- 3+ failures: "Não foi possível atualizar os dados após várias tentativas."

The banner is cleared on the next successful `loadData()`.

### Network failures — cloud save

`saveCloudState()` catches errors and leaves `hasPendingCloudChanges = true`. The cloud status indicator shows "alterações por guardar". A retry is scheduled in the `finally` block. The in-memory state is not rolled back — user edits are safe in localStorage even if the cloud save fails.

### Railway unavailable

- Cloud status indicator shows "Cloud: indisponível".
- The "Load Cloud" button will show an error alert.
- The "Run Settlement" button will fail and show the error message.
- All CSV data (bot picks, history) continues to load from GitHub raw URLs unaffected.
- Manual bets remain visible from localStorage.

### GitHub unavailable

- `loadData()` fails. CSV data is not refreshed. The fetch error banner is shown.
- `saveCloudState()` fails (Railway cannot reach GitHub). The save is queued for retry.
- The dashboard remains functional with stale data from the last successful load.

### Malformed data

- `parseDelimitedCsv()` handles BOM, mixed delimiters (auto-detects `;` vs `,`), and quoted fields.
- `normalizeBotCsvRow()` and `normalizeManualCsvRow()` apply field normalisation and defaults.
- Rows with missing `Data` or `Jogo` are filtered out after parsing.
- JSON parse errors in `loadLocalState()` default the affected field to `[]` or `{}`.

### Missing files

- If `picks_hoje_github.csv` is missing or empty, `state.footballDaily` is set to `[]`. The Daily Picks page shows no picks (no error).
- If `league_stats.csv` is missing, `state.leagueStats` remains `[]`. League Analytics shows no data.
- If `fixtures_today.csv` is missing, `state.fixtures` is `[]`. Scout kickoff lookup returns null; the bet can still be created without kickoff data.

## 14. UI Design Principles

**Single-page application, no framework.** The entire dashboard is one HTML file. There is no React, Vue, Angular, or any other UI framework. There is no npm, no bundler, no transpiler. A change to the frontend is a change to `index.html`.

**All JavaScript is inline.** The `<script>` block at the bottom of `<body>` contains all application logic. There are no external JS files. This makes the dashboard self-contained and deployable without a server.

**Dark theme by default.** The CSS uses custom properties (`--bg`, `--card`, `--text`, etc.) defined on `:root`. The palette is a dark navy (`#0a0f1e`) with blue accents (`#3b82f6`). Color-coded states: green (`--ok`) for wins, red (`--bad`) for losses, amber (`--warn`) for warnings, blue (`--info`) for neutral.

**Deterministic rendering.** Every render function reads from `state` and produces HTML. There is no two-way binding, no reactive system, no virtual DOM. Given the same `state`, the same HTML is always produced. This makes rendering predictable and debuggable.

**Mobile-responsive.** `isMobileDashboard()` checks viewport width (≤ 1024px). Mobile views use card layouts instead of tables. The sidebar collapses to a hamburger menu on mobile. `setActiveTab()` and `rerenderAll()` both account for the active viewport.

**PT-PT user interface.** All user-facing text is in Portuguese (Portugal). Field labels, alert messages, button text, and status indicators are all in Portuguese. Internal variable names and code comments are in English.

**Defensive rendering.** Every render call in `rerenderAll()` is wrapped in a try-catch with a `console.error` fallback. A JavaScript error in the Analytics panel does not crash the Daily Picks panel. Each section fails independently.

**Auto-save over manual save.** The user never explicitly clicks "Save". State mutations trigger `markDirty()`, which schedules a debounced auto-save. The cloud status indicator communicates the save state transparently. If the save fails, the indicator remains amber and a retry is attempted.

**No duplicated persistence.** `state.manualBets` is the single in-memory representation of manual bets. It is written to localStorage by `saveLocalState()` and to GitHub by `saveCloudState()`. There is no third copy. When the cloud is loaded, `state.manualBets` is replaced entirely — not merged.

**Inline Poisson model.** The Scout workspace runs the Poisson model in the browser in JavaScript. No round-trip to the backend is needed for analysis. The model uses `state.fixtures` (downloaded from GitHub raw URL) for λ values.
